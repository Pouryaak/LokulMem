<task type="auto">
  <name>Task 2: Create WorkerClient for request/response handling</name>
  <files>src/core/MessagePort.ts</files>
  <action>
    Create src/core/MessagePort.ts with WorkerClient class:

    1. Import types from ./Protocol.js, PortLike from ./types.js

    2. Export class WorkerClient:
       - private port: PortLike
       - private pendingRequests: Map<string, { resolve, reject, timeout }>
       - private messageHandler: ((message: WorkerMessage) => void) | null = null

    3. Constructor(port: PortLike):
       - Store port
       - Bind handleMessage and handleMessageError
       - Set up port.onmessage and port.onmessageerror

    4. Method request(type: string, payload: unknown, timeoutMs: number): Promise<unknown>
       - Generate id: crypto.randomUUID?.() ?? `${Date.now()}-${Math.random()}`
       - Return Promise that:
         - Sets timeout to reject after timeoutMs
         - Stores pending request in Map
         - Posts message with { id, type, payload }
       - On timeout: remove from Map, reject with error
       - NOTE: timeoutMs is required (no default); caller passes config.requestTimeoutMs ?? 5000

    5. Method onMessage(handler: (message: WorkerMessage) => void): void
       - Set messageHandler for non-response messages (like progress)

    6. Method terminate(): void
       - Clear all pending requests (reject with 'terminated')
       - Remove event listeners
       - Close port if possible

    7. Private handleMessage(event: MessageEvent<ResponseMessage | ProgressMessage>):
       - If message has id in pendingRequests: resolve/reject and cleanup
       - Else if messageHandler exists: call with message

    8. Private handleMessageError(event: MessageEvent):
       - console.error('Message deserialization error:', event)

    9. Export function createMainThreadPort(): PortLike
       - Returns a PortLike for main thread fallback
       - Creates a MessageChannel
       - One port is returned wrapped as PortLike, the other routes to in-memory handlers

    Per RESEARCH.md patterns:
    - Use request/response with UUID correlation
    - Handle messageerror events
    - Clean up timeouts to prevent memory leaks
  </action>
  <verify>grep -q "class WorkerClient" src/core/MessagePort.ts && grep -q "request.*Promise" src/core/MessagePort.ts && grep -q "createMainThreadPort" src/core/MessagePort.ts</verify>
  <done>WorkerClient class exists with request/response correlation, timeout handling, and main thread port creation</done>
